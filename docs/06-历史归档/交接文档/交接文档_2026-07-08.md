## 关键设计决策

1. **建筑属性从蓝图中推导**，不预设固定尺寸/价格/类型
   - 设施方块检测：`Blueprint.blockInfoAsList → instanceof AbstractBlockHut → ForgeRegistries.BLOCKS.getKey() → strip "blockhut" prefix`
   - 造价：遍历方块列表→查 `CValueRegistry.getBaseValueOrDefault()`→累加
   - 混合用途：检测到 ≥2 种不同类型的设施方块

2. **ZonePlanner 使用装箱算法而非固定格网**
   - 从 catalog 筛出的蓝图按面积降序排列
   - 大的优先填，塞到 density 覆盖率达标
   - 沿街面优先选 `isMixedUse()` 的蓝图
   - `filterByZoneType` 已从硬编码字符串改为 `CategoryMapper.getCategoriesForZoneType()`

3. **不依赖 ZoneData（已标记 @Deprecated）**
   - ZoneData 是旧的 7 字段持久化方案
   - 新字段 density/ownership 通过 `ZoneRect.toIntArray()/fromIntArray()` 序列化
   - 未来数据走 `CityState SavedData` 路线

## 已知问题

### 未修复

1. **ZoneBuilder 的点击事件注册** — `ZoneBuilder.tick()` 需要在 `ServerTickEvent` 中注册，`ZoneBuilder` 实例需要在 `MetroGenesis.java` 中创建并注册。当前代码仅实现了 tick 逻辑，**未接入事件总线**。

2. **ZonePlanner 调用入口未接入** — `ZonePlanner.plan()` 目前没有调用方。需要在 MayorBookScreen 或其他地方触发 `plan()` 并将结果传给 `ZoneBuilder.startGrowth()`。

3. **ZoneRect → ZoneData 数据流未打通** — 新 density/ownership 字段在 `ZoneRect.toIntArray()` 中有(9 int)，但 ZoneData 的 NBT 读写仍只处理 7 字段。如果后续要持久化到世界存档，需要更新 `ZoneData.save()/load()`。

4. **ZoneBuilder 的 C-Value 扣除用 TODO 标记** — `// TODO: 从国库扣除 C-Value` 处未实现。

### 设计层面

5. **CategoryMapper 中"商业"和"农业"没有对应的 mcCategory → MG 映射** — `getCategoriesForZoneType()` 虽然能返回 "商业" / "农业"，但当前 MineColonies 风格包中没有独立 "commercial" 或 "agriculture" 目录，所以不会有蓝图被归类为这两个分类。ZonePlanner 筛出来会为空。需要扩展方向：
   - (a) 添加新的 mcCategory 映射（如 "commercial" → "商业"）
   - (b) 或者 ZonePlanner 降级自动匹配到相近分类（如商业布局 → 生产建筑）

6. **ZoneRect.merge 的方向处理不完善** — 合并后取第一个区的 `direction`。建议后续增加根据合并后外框宽高比重算朝向的逻辑。
