## 3. Structurize 类加载陷阱

**现象**：`NoClassDefFoundError` 当加载继承了 Structurize 类的 MineColonies 类时。

**根因**：`Constants.MOD_ID` 若改为 `"metrogenesis"`，则 `@Mod.EventBusSubscriber(modid = Constants.MOD_ID)` 将 MineColonies 的事件发布到主 mod → Forge 扫描类 → 触发 Structurize 继承链 → 找不到类。

**当前方案**：`Constants.MOD_ID` 保持 `"minecolonies"`。事件绑定到 `"minecolonies"` mod（运行时不存在）→ Forge 静默跳过。

**长期方案**：逐步清理 MineColonies 源码中对 Structurize 的继承依赖，全部清完后改 MOD_ID。

## 4. 道路撤销系统架构

### 当前方案（2026-06-11）
每段铺路操作独享 `ChangeStorage`，精确记录每个方块变更。
- `roadUndoStack` / `roadRedoStack`（`Stack<ChangeStorage>`）
- Ctrl+Z → `storage.undo(serverLevel, null)`
- Ctrl+Y → `storage.redo(serverLevel)`

### 已知问题
1. **交互冲突**：MayorBookScreen 打开时 undo 后世界渲染不刷新
2. **漏块**：路基录制在实心层提前 break
3. **交叉路口干扰**：共享节点的路撤销后另一条接口断裂
4. **redo 局限**：不支持多段

### 理想方案（Phase 6+）
ChangeStorage 做录制，RoadData 做校验。撤销时先用 ChangeStorage 还原，然后 removeSegment + rebuildNode 修复交叉点。

## 5. 技术选型

| 决策 | 选择 | 原因 |
|------|------|------|
| 渲染引擎 | Vanilla shader + Forge RenderLevelStageEvent | 不依赖 OptiFine/Oculus |
| RTS 相机 | 独立 Entity | 非 Mixin 注入，兼容性好 |
| 旋转/镜像 | 复用 structurize RotationMirror | 不自己实现变换算法 |
| 撤销 | 复用 structurize ChangeStorage | 不重复造轮子 |
| 几何算法 | 参考 Effortless（不集成） | 只借鉴，不依赖 |
| 蓝图格式 | NBT palette + short[][][] | 自研，非 structurize 格式 |
| 资源加载 | 数据包路径 | MineColonies 原资源路径不变 |

## 7. 消化吸收策略

### 核心原则

> 以 MetroGenesis 自研 RTS 系统为核心主线 → 按需从 MineColonies 源码中提取模块 → 改姓（包名重命名）嵌入 → 逐步消化吸收

**不是直接调用殖民地 API，而是把殖民地的代码拿过来变成我们自己的。**

### 已消化模块

| 模块 | 原包名 | 新包名 | 核心能力 | 状态 |
|------|--------|--------|----------|------|
| **BlockUI** | `com.ldtteam.blockout` | `com.metrogenesis.blockui` | XML 声明式 UI、ScrollingList + DataProvider、20+ 控件 | ✅ 完成 |
| **Structurize** | `com.ldtteam.structurize` | `com.metrogenesis.structurize` | 5 阶段放置管线、Undo/Redo、Manager 操作队列 | ✅ 完成 |
| **Domum Ornamentum** | `com.ldtteam.domumornamentum` | `com.metrogenesis.domumornamentum` | 可材质化方块、动态 BakedModel 渲染 | ✅ 完成 |

### 待消化模块

| 模块 | 优先级 | 价值 | 吸收难度 |
|------|--------|------|----------|
| **IBuilding 等级体系** | P1 | 建筑 1→5 级升级，每级不同蓝图 | 中 |
| **请求系统** | P1 | 物品/服务自动调度 | 高 |
| **Colony 管理架构** | P2 | 15+ Manager 子系统，按需裁剪 | 高 |

### 消化方式

```
提取 MineColonies 某子系统 → 包名重命名 → 清理依赖 → 修改 API 适配 → 桥接到自研管线 → 该模块成为"我们的代码"
```
