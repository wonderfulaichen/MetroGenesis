## 1. 总体架构 & 代码规模

| 维度 | MineColonies | PolisCraft 当前 |
|------|-------------|----------------|
| 总代码量 | 数十万行（数百个子系统） | ~4,418 行（56 个 Java 文件） |
| 模块化 | 严格分层：API / core / common / client | 单模块，包级组织 |
| 依赖管理 | Gradle 多模块 + 外部依赖（Structurize/BlockUI） | 单模块，仅 Forge |
| 配置系统 | `AbstractConfig` 模块化配置基类 | 已迁移 `AbstractConfig` 但仅 1 个配置项 |
| 事件驱动 | ColonyManager → Colony → Citizen/job 逐级派发 | ColonyState.tick() 手动驱动 |
| 持久化 | 自定义 NBT + ChunkStorage 混合 | SavedData 单文件 |
| 多世界 | 原生支持多殖民地 | 单世界单 ColonyState |

**结论**：PolisCraft 是"原型阶段"的迷你镜像。架构骨架做对了（ColonyState → Manager 分层），但每个子系统的深度远不足以支撑游戏玩法。

## 2. AI 系统 ⭐ 最大差距

### PolisCraft 当前实现

**基类**：Minecraft 原版 `net.minecraft.world.entity.ai.goal.Goal`

**BuilderWorkGoal**（208 行）：
```java
private enum State { IDLE, CLAIM_SITE, MOVE_TO_SITE, PLACE_BLOCKS, COMPLETE }
```
- `canUse()` → 8 条扫描发现工地 → 进入 CLAIM_SITE
- `tick()` → 寻路完成 → 每 10 tick 放一个方块
- `doBuildStep()` → 从 BlueprintIterator 取下一个方块 → `level.setBlock()` 直接放置

**FarmerWorkGoal**（313 行）：
```java
private enum Mode { TEND_CROPS, WORK_AT_FACILITY, PLANT_SEEDS, TILL_LAND, GO_TO_VILLAGE, IDLE }
```
- 暴力扫描 16/64 半径 3 层嵌套找方块
- 按优先级：催熟 → 农场设施 → 播种 → 翻耕 → 找村庄

**MerchantWorkGoal**（210 行）：
```java
private enum Mode { AT_TOWNHALL, AT_VILLAGE, GOING_TO_VILLAGE, IDLE }
```
- 找市政厅/村庄钟 → 在旁边找空地 → 站桩撒粒子

### MineColonies 实现

```
BasicStateMachine<T, S extends AIWorkerState>
  → TickRateStateMachine
    → AbstractAISkeleton<T, S>
      → AbstractEntityAIInteract
        → AbstractEntityAIStructure<J, S>
          → EntityAIStructureBuilder
```

**核心差异**：

| 特性 | PolisCraft | MineColonies |
|------|-----------|-------------|
| 状态机 | 简单枚举 + if-else | `BasicStateMachine` 通用框架 |
| Tick 优先级 | 无 | 4 层：aiBlocking > event > stateBlocking > currentState |
| Tick Rate 控制 | 无 | 每个状态可配 `getTickRate()` |
| 事件驱动 | 无 | `eventTransitionMap` 根据事件切换状态 |
| 路径重试 | 无 | 超时重试 + 不同模式 |
| 进食中断 | 无 | `okayToEat` 参数控制 |
| Bucket 分桶系统 | **缺失** | 材料按背包容量分桶，一桶一桶运 |
| 放置处理器链 | `level.setBlock()` 直接 | 18 种处理器（连接状态/TileEntity/氧化/感染/水...） |
| 动画 | 无 | 手臂/工具摆动动画同步 |
| 进度可视化 | 简单的 % | `BuildingProgressStage` + 施工围栏全彩进度 |

**差距等级**：🔴 极大 — 这是目前最明显影响游戏体验的部分。

### 子差距：放置系统

| 对比项 | PolisCraft | MineColonies |
|--------|-----------|-------------|
| 放置方式 | `level.setBlock(worldPos, state, 3)` | 18 Handler 责任链 |
| 特殊方块 | 完全不管 | 床/门/梯子/告示牌/箱子/活塞/红石... |
| TileEntity | 丢失数据 | `setBlockState` + 单独 `setTileEntityData` |
| 连接状态 | 无 | 自动处理梯子支撑/栅栏连接/门朝向 |
| 水管理 | 无 | 先排水后放方块 |
| 氧化/风化 | 无 | 自动检查邻居状态 |
| 旋转 | 无 | `MIRROR`/`ROTATE` 策略 |

## 3. Job / 职业体系

### PolisCraft 当前

```java
// ModJobs.java (79 行)
private static final Map<String, Function<Mob, Goal>> REGISTRY = new HashMap<>();
static {
    register("farmer",   FarmerWorkGoal::new);
    register("builder",  BuilderWorkGoal::new);
    register("merchant", MerchantWorkGoal::new);
}
```

- 每个职业 = 一个 Goal 实例，由 `CitizenEvents` 在生成时添加到实体
- 3 个职业，每个约 200-300 行

### MineColonies 实现

```
AbstractJob<AI extends AbstractAISkeleton<J, S>, J extends AbstractJob<AI, J>>
  → AbstractJobStructure<AI, J>
    → JobBuilder
    → JobMiner
    → JobLumberjack
    → ... (15+ 职业)
```

| 特性 | PolisCraft | MineColonies |
|------|-----------|-------------|
| 职业数量 | 3 | 15+ |
| 注册 | Map<ID, factory> | BuildingEntry 桥梁（自引用泛型） |
| Job ↔ AI 关联 | Goal 直接引用 | Job 持有 AI 引用，Job 管理 asyncRequests |
| 异步请求 | 无 | `asyncRequests` + Token 回调 |
| 工作模式 | 无 | AUTO/MANUAL/StringModeSetting |
| 升级 | 无 | 建筑等级 → 解锁新功能 |
| 动作日志 | `LOGGER.debug` | `actionsDone` 列表（发给 GUI） |

**差距等级**：🔴 极大 — 当前 Map 方案只适合原型验证

## 4. 建筑注册系统

### PolisCraft 当前

```java
// BuildingType.java (88 行)
public static final BuildingType TOWN_HALL  = new BuildingType("town_hall", ...);
public static final BuildingType FARM       = new BuildingType("farm_facility", ...);
public static final BuildingType HOUSE      = new BuildingType("house", ...);
public static final BuildingType WORKSHOP   = new BuildingType("workshop", ...);
public static final BuildingType WAREHOUSE  = new BuildingType("warehouse", ...);
public static final List<BuildingType> ALL  = List.of(...);
```

```java
// PolisCraft.java 注册 7 个方块
BLOCKS.register("town_hall",   TownHallBlock::new);
BLOCKS.register("farm_facility", ...);
BLOCKS.register("construction_marker", ...);
BLOCKS.register("construction_tape", ...);
BLOCKS.register("house", ...);
BLOCKS.register("workshop", ...);
BLOCKS.register("warehouse", ...);
```

### MineColonies 实现

```
BuildingEntry
├── BuildingEntry-Citizen (JobBuilder, JobFarmer...)
├── BuildingEntry-Crafter (各工匠)
├── BuildingEntry-Residential
└── BuildingEntry-Military
```

关键差异：
- **三层继承**：`AbstractBuilding<B extends AbstractBuilding<B>> → AbstractBuildingWorker<B, J> → 具体建筑`
- **双重注册**：方块注册 + BuildingEntry 桥梁 = 解耦
- **ModuleProducer**：每个建筑可以组合多个模块（ModuleInventory/ModuleWorker/ModuleUpgrade...）
- **属性驱动**：JSON 配置建筑参数（速度、容量、需求）

| 特性 | PolisCraft | MineColonies |
|------|-----------|-------------|
| 建筑数量 | 5（硬编码） | 30+（+ 动态注册） |
| 升级系统 | 无 | 5 级，每级解锁新功能 |
| 模块组合 | 无 | ModuleProducer 模式 |
| 参数定义 | Java 硬编码 | JSON data pack |
| 蓝图路径 | 硬编码字符串 | `getBlueprintPath()` 按等级返回 |
| 工人分配 | 有（简单） | BuildingEntry + 条件判定 |
| GUI 生成 | 手动写 Screen | 自动生成（BlockUI XML） |

**差距等级**：🟡 中 — 5 种建筑做原型够了，但扩展性差

## 5. 市民数据系统

### PolisCraft 当前

**核心类**：`citizen.types.CitizenData`（233 行）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int | 市民 ID |
| uuid | UUID | 全局唯一 ID |
| name | String | 生成的名字 |
| female | boolean | 性别 |
| textureId | int | 纹理 ID |
| textureSuffix | String | "_a", "_b", "_d", "_w" |
| job | String | 职业 ID |
| wallet | int | 钱包余额 |
| satisfaction | int | 0-100，满意度 |
| isDirty | boolean | 脏标记 |

**管理器**：`CitizenManager` — 持有 `Map<Integer, CitizenData>`
- 创建/移除/ID 复用/性别平衡
- tick 驱动

**特点**：
- JSON 资源包驱动名字生成（中西文双支持）
- 满意度 → 工作效率映射（<30: 0.5x, <60: 0.8x, >80: 1.2x）
- 工资计算 switch-case

### MineColonies 实现

```
CitizenData
├── id / UUID / name / texture
├── skills: Map<SkillType, Skill>    ← 关键差异
├── workStats: workTime / efficiency / productionCount
├── inventory: 15 格背包
├── statusMessages: 活跃状态文本
├── health / saturation
├── job / building / buildingLevel
├── happiness: 受 6+ 因素影响的食物/社交/健康
```

| 特性 | PolisCraft | MineColonies |
|------|-----------|-------------|
| 技能系统 | **缺失** | 9 种技能，按工作+建筑等级增长 |
| 背包 | **缺失** | 15 格库存 |
| 背包容量 | **缺失** | Strength 技能影响 |
| 饥饿度 | **缺失** | 食物系统 |
| 幸福感因子 | 仅满意度一项 | 6+ 因子（食物/休息/健康/房屋/社交/等级） |
| 状态消息 | **缺失** | IK StatusMessages |
| 职业经验 | **缺失** | Skill XP + Level |
| 纹理多样性 | textureId + suffix | 多层级：职业/等级/性别/自定义 |
| NBT 持久化 | ✔ | ✔ |

**差距等级**：🟡 中 — 数据结构基础不错，但缺少技能/背包/食物三大支柱

## 6. 施工 & 蓝图系统

### PolisCraft 当前

**施工管理器**：`ConstructionManager`（122 行）
- `Map<Integer, ConstructionSite>` 管理工地
- 创建/查找/移除/tick

**施工工地**：`ConstructionSite`（200 行）
- 位置 + 建筑类型 + Zone + 进度
- `complete()` → 移除围栏 → 替换方块 → 触发 Colony 初始化
- 市政厅完成时刷出 3 个市民

**全息蓝图**：自研 `BlueprintRenderer`（VertexBuffer + ChunkBufferBuilderPack）
- 全息图网络同步：`HologramSyncPacket`
- 渲染分层：solid → cutoutMipped → cutout → translucent

**蓝图迭代器**：`BlueprintIterator` — 逐块遍历

### MineColonies 实现

```
StructurePlacer    → 蓝图加载/放置
StructureIterator  → 进度迭代（支持暂停/跳跃）
StructureManager   → 全局蓝图管理
```

| 特性 | PolisCraft | MineColonies |
|------|-----------|-------------|
| 蓝图格式 | 自研 short[] palette | Structurize 标准格式（含 NBT） |
| 蓝图迭代 | 简单 next() | 支持暂停/恢复/设置进度 |
| 建筑等级蓝图 | **缺失** | 每级独立蓝图 |
| 材料需求计算 | **缺失** | 4 阶段遍历（SOLID→WEAK→DECO→ENTITIES） |
| 库存检查 | **缺失** | Builder 检查背包 + 仓库 |
| 自动补料 | **缺失** | Request System 自动配送 |
| 损坏修复 | **缺失** | Repair 模式 |
| 旋转/镜像 | **缺失** | MIRROR/ROTATE 策略 |
| 全息图着色 | 已完成 | 已完成 |
| 围栏系统 | Zone + particle | ConstructionTapeHelper + dynamic rendering |

**差距等级**：🟡 中 — 有全息图渲染自研成果，但材料/补料/多等级蓝图属于缺乏的

## 8. 请求系统 & 物流 ⭐ 第二大差距

### PolisCraft 当前

**完全缺失**。

当前 Builder 工作流程：
```
Builder 发现工地 → 走到工地 → level.setBlock() 逐块放置
```
- 不需要材料 — 直接虚空造块（creative 模式本质）
- 没有物流 — 没有 Delliveryman 职业
- 没有库存 — Builder 不检查背包
- 没有 Request → Fulfill 闭环

### MineColonies 实现

**Request System 核心**：

```
AbstractRequestingEntity
  → hasRequest()
  → createRequest() → 返回 Token
  → onRequestResolved() → 回调
```

**Token 驱动 5 态状态机**：
```
CREATED → IN_TRANSIT → RESOLVED
                ↓ (失败)
              CANCELLED
```

**解析器责任链**（9 级，按优先级）：
1. `PlayerRequestResolver` — 玩家仓库/背包
2. `BuildingRequestResolver` — 同建筑其他模块
3. `WarehouseRequestResolver` — 中央仓库
4. `CitizenRequestResolver` — 其他市民背包
5. `DeliverymanRequestResolver` — 配送员任务
6. `RemoteWarehouseResolver` — 远程仓库
7. `MineRequestResolver` — 矿井产出
8. `PlantationResolver` — 种植园
9. `CreativeRequestResolver` — 创造模式兜底

**配送员 AI**：
- 评分排序（距离 × 紧急性 × 仓库容量）
- 批量合并（同一路径上的多个任务）
- 老化优先级（等待越久分越高）

**差距等级**：🔴 缺失 — 这是从"原型"到"可玩游戏"最大的结构性缺口

## 12. 重构优先级建议

### 🥇 P0 — 必做，否则游戏不成立

**① 请求系统** — 这是 MineColonies 的中枢神经系统
- 如果没有 Request System，Builder 只能虚空造块（像现在这样）
- 需要：Token → Resolver Chain → Delivery → Fulfill
- 参考：§5.1-5.5 的 Token 驱动 + 状态机 + 解析器责任链

**② AI 状态机** — 所有职业的基础设施
- 当前 `Goal.canUse()` 太简单，不支持复杂行为
- 不需要 1:1 复制 MineColonies，但要引入 `TickRateStateMachine` 核心模式
- 参考：§6.8 的 BasicStateMachine → TickRateStateMachine 模式

### 🥇 P1 — 核心玩法需要

**③ 物流/配送员** — 和 Request System 配合使用
- 没有配送员，材料不会从 A 到 B
- 可作为 P0 之后立即接手的模块

**④ 放置处理器链** — Builder 的工作入口
- 当前 `level.setBlock()` 在某些 Minecraft 版本会丢失数据
- 至少实现：基础方块 + TileEntity + 门/床/方向

## 重构路线图（建议）

```

  └─ 放置处理器链（门/床/红石/TE）

  └─ 市民技能 + 背包 + 饥饿

第 4 阶段（完整化）：
  └─ 15+ 职业
  └─ 完整物流网络
```
