## 一、🔴 阻断级问题（玩家无法继续或数据损坏）

### 1.1 招募费用客户端无状态禁用 — 资金不足时按钮仍可点击

> 勘误：之前误判为"服务端无校验"。经查 `ColonyState.spend()`（第87-94行）和 `TownHallMenu.clickMenuButton()`（第164-168行）在服务端做了完整的余额检查。但客户端交互仍然存在严重问题。

**触发场景**：玩家国库 < 100 C-Value 时点击招募按钮。

**代码分析**：
- ✅ 服务端：`ColonyState.spend(int amount)` 执行 `if (funds >= amount)` 检查，不足时返回 false；`clickMenuButton()` 正确使用 `if (!colony.spend(100))` 阻止招募并发送提示消息
- ❌ 客户端：`TownHallScreen.java` 中的三个招募按钮是固定渲染的 `drawButton()`，**未根据 `funds < 100` 设为禁用状态**。玩家点击后得到一个服务端"国库不足"的聊天栏消息，缺乏 GUi 层面的即时反馈
- ❌ 代码冗余：第181行 `colony.addToTreasury(0)` 是空操作（加0），疑似笔误或桩代码残留

**影响评估**：
- 玩家体验：点击招募无反应 → 收到服务端拒绝的 chat 消息 → 困惑：按钮明明是亮的为什么点不了？
- 无破坏性漏洞（服务端有安全守卫），但交互体验差
- `addToTreasury(0)` 残留代码可能引发后续维护困惑

**改进建议**：
- 客户端 `TownHallScreen.render()` 中根据 `menu.getFunds() < 100` 渲染灰色禁用按钮 + 悬停显示"资金不足"
- 移除 `colony.addToTreasury(0)` 的冗余行
- 考虑客户端即时反馈：资金不足时按钮文字改为"§8资金不足"而非"招募"

### 1.2 施工管理页面只有两个硬编码建筑选项

**触发场景**：玩家放置施工标记 → 打开 ConstructionMarkerScreen。

**代码分析**：`ConstructionMarkerScreen.java` 第18-20行硬编码 `SEL_FARM_Y=50`（农场）和 `SEL_HALL_Y=80`（市政厅），`mouseClicked()` 仅响应对应 buttonId 0 和 1。建筑类型在源码中写死，未引用 `BuildingType.ALL` 或注册表。

**影响评估**：
- 玩家放置施工标记后只能建造农场或市政厅，与该模组"市民自治、五个政府部门、产业链"的设计规模严重不匹配
- 施工标记 `ConstructionMarkerBlock` 的前置逻辑也仅有这两个选项，玩家无法扩展城市

**改进建议**：
- 改造为动态建筑列表，参考 `BuildToolScreen.java` 加载 `BuildingType.ALL` 的方式
- 或将施工管理与建筑选择工具合并，避免功能割裂

### 1.3 Guilded/MayorBookScreen 缺失建筑放置确认步骤

**触发场景**：市长管理书 → 选择建筑类型 → 点击地面。

**代码分析**：`MayorBookScreen.java` 包含俯视视角和建筑分类工具栏，但在 `mouseClicked()` 中将点击坐标直接转换为建筑放置位置后立即发送 `BuildToolPlacementMessage`。**无二次确认弹窗**。对比 `BuildToolScreen.java` 有独立的列表选择和确认按钮流程。

**影响评估**：误触即放置，无撤销机制。市民已在建造中的建筑不可取消，造成资源永久损失。

**改进建议**：
- 市长管理书增加点击放置前的确认弹窗（"确定在此位置建造 [建筑名称]？"）
- 或切换到建造预览模式（半透明轮廓），按右键取消、左键确认

### 1.4 客户端与服务端之间无建造权限校验

**触发场景**：两个玩家同时在同一坐标附近放置建筑。

**代码分析**：`BuildToolPlacementMessage` 和 `CaptureBlueprintPacket` 均通过 `NetworkHandler.CHANNEL.sendToServer()` 发送。服务端处理消息的 handler 未读取（但根据设计推测需校验包含关系）。当前代码层面，**服务端 handler 未显式校验该坐标是否已属于其他区块/建筑**。

**影响评估**：区块重叠 → 两个建筑的市民 AI 路径冲突 → 市民卡在方块中 → 服务端 tick 性能下降。

**改进建议**：
- 服务端 handler 增加区块占用检测：`if (buildingManager.isOccupied(pos)) return;`
- 客户端放置时预览红色碰撞指示

## 二、🟡 困惑级问题（玩家困惑、学习成本高或行为异常）

### 2.1 编码问题 — 中文显示为 Unicode 转义序列/乱码

**触发场景**：玩家打开任何 GUI 界面。

**具体文件**：
- `TownHallScreen.java`：`\u5e02\u653f\u5385` → "市政厅" 本应正常显示，但源码文件编码非 UTF-8
- `ConstructionMarkerScreen.java`：`\u00a76\u00a7l\u65bd\u5de5\u7ba1\u7406` 等转义序列
- 所有 Screen 文件均有中文注释乱码

**影响评估**：游戏内 GUI 标题和提示文字显示为乱码方块（????），玩家看不懂界面。对于非中文环境的玩家（UTF-8 编码），问题更严重。

**改进建议**：
- 统一所有 `.java` 源文件为 UTF-8 编码
- 所有玩家可见字符串应使用 `Component.translatable()` 配合语言文件，而非硬编码

### 2.2 招募费用固定 100 C-Value — 缺乏动态定价

**触发场景**：玩家反复招募同一职业。

**代码分析**：`TownHallMenu.java` 中三个按钮的招募费用均为硬编码 `colony.spend(100)`。`CitizenData.java` 的基础薪水 farmer=15, builder=20, merchant=18 各不相同，但招募费不分职业。

**影响评估**：
- 商人招募费 100 但日薪 18（回本需 5.6 天），农夫招募费 100 但日薪 15（回本需 6.7 天）—— 商人性价比显著高于农夫，玩家会只招商人
- 无市场调节机制：市民数量增加应降低招募费（供给增加）而非固定

**改进建议**：
- 招募费应基于职业生产力：`recruitCost = baseCost × (1 + currentCitizens / maxCitizens)`
- 不同职业应有差异化招募费：
  - 农夫：50 C（低技能门槛）
  - 建造师：150 C（高技能要求）
  - 商人：200 C（高回报职业）

### 2.3 职业切换按钮编码溢出风险

**触发场景**：城市发展到大后期（citizenId > 214748364）。

**代码分析**：`TownHallMenu.java` 使用 `buttonId = 1000 + citizenId * 10 + jobIdx` 编码职业切换。`citizenId` 为 `int` 类型。当 `citizenId > 214748364`（约 2.1 亿）时，`citizenId * 10` 溢出为负数，导致 buttonId 计算错误。

**影响评估**：虽然在正常游戏规模（单城市几百人）下不会触发，但多世界服务器或长期运营场景存在理论风险。

**改进建议**：使用 `long` 计算：`long btnId = 1000L + (long) citizenId * 10 + jobIdx`，或在 Container 端使用自定义封包而非 buttonId 编码。

### 2.4 满意度进度条颜色分类缺乏告知

**触发场景**：查看市民列表中满意度进度条。

**代码分析**：`TownHallScreen.java` 中满意度进度条颜色硬编码为绿(≥80)、黄(≥50)、红(<50)。但 **GUI 上未标注阈值说明**。

**影响评估**：玩家看到红色的进度条但不知道原因（饥饿？没工作？住所不够？），也不知道怎么改善。设计文档中的市民满意度和工作效率公式（`getWorkEfficiency()` 方法）玩家在游戏内无从得知。

**改进建议**：
- 在市民详情页面增加满意度影响因素列表：
  - +10 有工作
  - +5 有住所
  - -15 饥饿
  - -20 无工作
- 悬停进度条显示 tooltip 说明阈值

### 2.5 建筑放置无预览 + C-Value 死锁

**触发场景**：选择建筑后点击地面放置。

**代码分析**：`BuildToolScreen.java` 的 `confirmSelection()` 直接发送网络封包并关闭界面。`BuildToolPlacementMessage` 未包含预览阶段。

**影响评估**：
- 16×16 区块建筑不同于单方块，原版 MC 的半透明方块预览逻辑不适用于大区块建筑
- 直接放置可能因坐标偏差导致建筑错位，且无撤销机制
- 与 mc-dev-advisor 确认结论：建议采用 **2D 区块网格预览**（轻量级，与区块设计对齐）而非 3D 半透明渲染

**改进建议**：
- 选建筑后在地面渲染 2×2/4×4 区块网格线框，标记该建筑的占地面积
- 复用 `BuildingManager.isOccupied()` 做占用检测：绿色=可放置，红色=冲突
- 点击地面进行二次确认"确定在此建造？"

### 2.6 蓝图扫描入口方向计算逻辑偏差

**触发场景**：拖拽扫描较大区域（对角线方向）。

**代码分析**：`BlueprintCaptureResultScreen.getEntryDirection()` 使用 `Math.abs(dx) >= Math.abs(dz)` 判断方向。当玩家从东北向往西南方向拖拽时，入口方向判断为主轴向（dx 负值→"西"），但实际上入口应为拖拽的"结束端"。

**影响评估**：扫描结构放置到世界后方向与预期不符，玩家需要手动旋转调整，增加学习成本。

**改进建议**：
- 入口应定义为拖拽起始点所在的平面方向
- 或增加玩家视角方向辅助判断：`player.getDirection()`

### 2.7 3D 预览渲染性能隐患

**触发场景**：扫描大区域结构（>50³）。

**代码分析**：`BlueprintCaptureResultScreen.render3DPreview()` 在体素模式下三层嵌套循环 (`for by..bz..bx`) 遍历所有非空方块。每个方块调用 `getMapColor()`，异常处理中仅 `catch (Exception e) {}` — **静默吞异常**。

**影响评估**：
- 100³ 区域（100 万方块）在体素模式下需要 100 万次 `getMapColor()` 调用和等轴测投影运算，单帧渲染可能 >50ms
- 异常静默吞掉，调试困难
- `BlueptintUtil.scanRegion()` 若未缓存，每次旋转都重新扫描（小区域 <50³），但大区域使用原地变换后，`render3DPreview` 的遍历逻辑不变

**改进建议**：
- 增加 LOD（细节层次）：远距离方块采样（隔2取1）或合并同色块
- 使用 `MapColorCache` 缓存颜色值
- 异常应记录日志而非静默吞掉
- 体素预览增加帧率计数器，超时自动降级为线框

## 三、🔵 建议级（体验优化、设计建议）

### 3.1 按ESC关闭蓝图预览会丢失未保存的蓝图

**触发场景**：扫描建筑后按 ESC 关闭。

**代码分析**：`onDiscard()` 调用 `onClose()` 直接退出。ESC 键默认行为也是 `onClose()`。玩家可能误触 ESC 丢失扫描结果，无"未保存"提示。

**建议**：
- 在 `onClose()` 中检查是否已保存：`if (!saved) confirmDiscard()`
- 或增加 ESC 拦截：弹出确认框"蓝图尚未保存，确定丢弃？"

### 3.2 BuildToolScreen 中确认/取消按钮易误触

**触发场景**：选择建筑后点击确认。

**代码分析**：确认和取消按钮位置在屏幕底部中间（`btnCenterX ± 8`），间距仅 16px。标准 Minecraft GUI 习惯"右键取消、左键确认"，此处两按钮间距过小。

**建议**：
- 增大按钮间距至至少 40px
- 或遵循 MC 惯例：右下为确认、左下为取消
- 增加"确认放置"二次确认弹窗

### 3.3 市政厅 GUI 无分页导航

**触发场景**：城市人口 > 10（市民列表超出一屏）。

**代码分析**：`TownHallScreen.java` 的市民列表使用简单的垂直布局，无滚动条和分页。当前硬编码 Y 坐标从 30 开始每 20px 一个条目，约可容纳 10 人。

**建议**：
- 增加 `ScrollingList` 或换行分页
- 增加搜索/筛选（按职业、满意度排序）
- 参考 `WindowCitizenPage.java` 的分页设计

### 3.4 市民详情页中无返回市政厅按钮

**触发场景**：从市政厅点击市民名称 → CitizenDetailScreen。

**代码分析**：`CitizenDetailScreen.java` 打开后没有返回市政厅的按钮，只能按 ESC 关闭。玩家要查看另一个市民需重新打开市政厅。

**建议**：
- 添加"返回市政厅"按钮
- 或实现上一个/下一个市民的导航箭头

### 3.5 C-Value 经济系统与 Minecraft 原版物品体系缺乏桥接 — 死锁问题（升为 🟡）

> 基于 mc-dev-advisor 的深入分析修正。

**触发场景**：玩家首次打开市政厅，国库 = 0 C-Value。

**代码分析**：设计文档定义 C-Value = 基准价值 × (1 + 科技系数) × 供需因子。招募费硬编码 100 C-Value。但玩家初始国库为 0，且 **无任何初始获取 C-Value 的渠道**。

**这是"先有鸡还是先有蛋"的死锁**：
- 玩家需招募市民才能产生经济活动（税收/贸易）
- 但招募市民需要 C-Value
- 没有市民就没有 C-Value 来源
- 没有 C-Value 就无法招募市民

**与 mc-dev-advisor 确认结论**：
- 首选方案：**启动物资箱**（类似 MineColonies 的补给物资，内含启动资源）
- 次选方案：**原版物品捐献机制** — 玩家向市政厅捐赠原版物品，按价值表折算为 C-Value。可复用原版村民交易 `TradeOffer` 数据结构做价值映射
- 无论哪种方案，均需在玩家第一个市政厅放置时自动触发

**建议**（综合后）：
- 市政厅放置时赠送启动物资箱（含 500 C-Value+基础建材）
- 或增加捐赠功能，将原版物品兑换为 C-Value（如 1 钻石 = 50 C）

### 3.6 城市报表/统计面板缺失

**触发场景**：玩家想了解城市经济运行情况。

**现状**：当前 GUI 仅展示国库总额和人口数字。无收入/支出明细、产业链地图、C-Value 流通追踪。

**建议**（基于设计文档中的五大部门体系）：
- 财政面板：收入明细（税收/贸易/其他）、支出明细（工资/建设/维护）
- 人口面板：市民结构（职业分布/满意度分布/年龄结构）
- 贸易面板：进出口量、热门商品、C-Value 汇率
- 这些数据可通过 `ContainerData` 同步到客户端

### 3.7 市长管理书（MayorBookScreen）的快捷键缺乏引导

**触发场景**：首次打开市长管理书。

**现状**：WASD 移动、右键旋转、滚轮缩放。但 GUI 上无操作提示，新玩家茫然。

**建议**：
- 首次打开时显示 3 秒操作引导覆盖层
- 底部状态栏增加小字快捷键提示：`WASD移动 | 右键旋转 | 滚轮缩放`
- 右上角增加"?"帮助按钮

### 3.8 施工进度条无预估完成时间

**触发场景**：查看施工进度。

**代码分析**：`ConstructionMarkerScreen.java` 显示进度百分比但无剩余时间。建造师 AI 的工作效率受 `getWorkEfficiency()` 影响（满意度驱动），但玩家看不见效率倍率。

**建议**：
- 增加估计剩余时间：`estimatedTicks = remainingBlocks / (builderCount × efficiency)`
- 进度旁显示"效率：×1.2（满意）"或"效率：×0.5（不满）"

### 3.9 无实体物流网络缺乏反馈

> 基于 mc-dev-advisor 的成本-收益分析修正。

**触发场景**：城市物流系统运转。

**现状**：设计文档描述"无实体物流网络"基于道路网格执行最小费用最大流算法，每日凌晨批量运输。但游戏中无任何物流路径可视化。

**与 mc-dev-advisor 确认结论**：
- "无实体物流"是设计选择（性能优势），"无反馈"是体验失败
- 推荐最短开发路径：**粒子效果轨迹**（1-2 天可完成）
- 物流看板功能（商务部门功能）推迟到 Phase 3
- 调试网络视图成本高，建议搁置

**建议**（综合后）：
- 近期（Phase 1-2）：在每日凌晨物流执行时，沿活跃路径生成蓝色粒子效果，持续数秒
- 中期（Phase 3）：商务部门页面增加"今日吞吐量"统计 + 热门商品运输路线示意
- 远期：调试网络视图（按住 F3 类组合键显示流量热力图）

## 四、玩家第一印象总评

### 首次体验模拟

*（模拟一个从未接触过 PolisCraft 的 MC 玩家首次进入游戏）*

1. **打开游戏 → 看到"MetroGenesis"加载界面**。OK, 新模组。
2. **合成市政厅 → 放置 → 右键打开**。GUI 显示"市政厅"，中文乱码或者 `\u` 编码显示。困惑。
3. **看到三个招募按钮：农夫/建造师/商人，每个 100 C-Value**。我只有 0 C-Value。点不了。卡住了。怎么获得 C-Value？**没有任何引导**。
4. **捡起地上的石头 → 尝试对着市政厅右键**。没反应。怎么办？
5. *（去查维基/源码）* 哦，需要先造一个"初始设施"... 但是施工标记只能造农场和市政厅？那市政厅已经有了，只缺农场。
6. **建造农场 → 等待建造师施工**。进度条 32%... 进度条 67%... 完成！农夫招募成功！
7. **点击农夫 → 看 CitizenDetailScreen**。满意度和职业分配按钮。满意度 50（黄色）。看不懂为什么 50。
8. **打开市长管理书**。哇，这个界面好看！WASD 移动...ESC 退出... 等等，怎么用？放在那里建造？点了没反应？
9. *（误触放置了一个建筑）* 哦不，放错位置了！不能撤销吗？

### 核心体验结论

**PolisCraft 的设计理念（C-Value 经济、市民自治、区块规划）非常出色，但当前的交互实现将大多数玩家挡在了学习门槛前。**

- **亮点**：玻璃态的 MayorBookScreen 设计（领先于大部分 MC 模组的视觉水平）、等轴测蓝图预览的工程创新、C-Value 经济体系的深度
- **痛点**：零新手引导、中文编码问题、关键交互路径断裂（特别是 C-Value 获取→招募的因果链不明）
- **风险**：招募无余额校验（贯穿经济系统）、建筑放置无校验（多人模式灾难）、施工选项过少（城市发展受限）

### 优先级建议（按修复紧迫性）

1. **🔴 客户端招募按钮增加资金不足禁用状态** — 服务端有校验，但客户端按钮始终亮着，误导玩家
2. **🔴 施工管理动态建筑列表** — 否则城市无法发展
3. **🟡 解决 C-Value 死锁**（启动物资箱/物品捐献）— 否则玩家无法开始游戏循环
4. **🟡 修复中文编码** — 否则所有中文玩家看到乱码
5. **🔵 添加新手引导** — 市长管理书操作提示、满意度因素说明
6. **🔴 建筑放置增加权限校验** — 多人模式必备
7. **🟡 职业招募差异化定价** — 平衡性优化
8. **🔵 2D 区块网格放置预览** — 替代原版 MC 预览逻辑
9. **🔵 粒子效果物流轨迹** — 最低视觉反馈
